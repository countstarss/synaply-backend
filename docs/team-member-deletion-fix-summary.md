# 团队成员删除功能修复总结

## 问题概述

在删除团队成员时，遇到了多个外键约束错误，导致删除操作失败。

## 问题原因分析

### 1. 根本原因

**数据库外键约束设计问题**：
- 多个表通过外键引用了 `team_members` 表
- 这些外键约束使用了 `ON DELETE RESTRICT`，意味着必须先删除或处理所有引用记录，才能删除 `TeamMember`
- 初始实现中**没有考虑级联删除或数据清理**，直接尝试删除 `TeamMember`，导致外键约束冲突

### 2. 为什么会出现这种情况？

#### 设计层面的问题：
1. **缺乏统一的删除策略**：
   - 没有在 schema 层面定义统一的 `ON DELETE CASCADE` 或 `ON DELETE SET NULL` 策略
   - 不同表使用了不同的删除策略（RESTRICT、SET NULL、CASCADE），导致处理逻辑复杂

2. **业务逻辑不完整**：
   - 删除团队成员是一个**复杂的业务操作**，不仅仅是简单的数据库删除
   - 需要考虑：
     - 该成员创建的内容如何处理（评论、问题、工作流等）
     - 该成员被分配的任务如何处理
     - 该成员的所有权如何转移
     - 历史记录如何保留

3. **Schema 与数据库不一致**：
   - `notifications` 表在数据库中存在，但在 `schema.prisma` 中没有定义
   - 导致代码中无法通过 Prisma 客户端访问这些关系，需要使用原始 SQL

#### 开发流程问题：
1. **缺乏完整的外键约束检查**：
   - 没有在实现删除功能前，系统性地检查所有引用 `team_members` 的表
   - 采用"遇到错误再修复"的被动方式，而不是主动预防

2. **测试不充分**：
   - 没有测试删除有大量关联数据的成员的情况
   - 没有覆盖所有可能的外键约束场景

#### ✅ Issue.creatorId  
- **创建迁移**：`20250627125340_init_workflow` - `issues_creator_id_fkey`
- **删除迁移**：`20250710022031_simplify_issue` - 已删除约束
- **状态**：约束已不存在，无需处理

### 2. 业务逻辑考虑不足

#### 数据完整性问题：
- **评论删除**：删除成员时，该成员创建的所有评论都被删除，这可能不是期望的行为
  - 更好的做法：保留评论，但标记作者为"已删除成员"
  
- **历史记录丢失**：删除 `IssueActivity` 和 `IssueStepRecord` 会导致历史记录丢失
  - 更好的做法：保留记录，但标记操作者为"已删除成员"

#### 所有权转移问题：
- **工作流转移**：当前逻辑是转移给其他管理员，但如果该成员是唯一的管理员且创建了工作流，会阻止删除
  - 更好的做法：提供选项让用户选择如何处理（转移、删除、或阻止删除）

#### 通知处理：
- **通知删除**：删除成员时，该成员的所有通知都被删除
  - 更好的做法：可能需要保留通知记录用于审计

### 改进建议

1. **统一删除策略**：
   - 在 schema 层面统一使用 `ON DELETE CASCADE` 或 `ON DELETE SET NULL`
   - 或者创建数据库函数/触发器统一处理

2. **完善 Schema 定义**：
   - 将 `notifications` 表添加到 `schema.prisma`
   - 明确 `Project.creatorId` 和 `Issue.creatorId` 的关系定义
   - 确保 schema 与数据库一致

3. **改进业务逻辑**：
   - 提供"软删除"选项（标记为已删除，但保留数据）
   - 提供更灵活的所有权转移策略
   - 保留历史记录，但标记操作者状态

4. **完善测试**：
   - 创建测试用例覆盖所有外键约束场景
   - 测试删除有大量关联数据的成员
   - 测试边界情况（如唯一管理员、唯一拥有者等）

5. **文档化**：
   - 记录所有引用 `team_members` 的表和字段
   - 记录删除操作的完整流程
   - 记录业务规则和决策

## 经验教训

1. **设计阶段就要考虑删除策略**：
   - 在设计数据库 schema 时，就应该考虑删除操作的复杂性
   - 统一使用合适的 `ON DELETE` 策略

2. **主动检查而非被动修复**：
   - 在实现删除功能前，应该系统性地检查所有外键约束
   - 使用数据库查询工具检查所有引用关系

3. **Schema 与数据库保持一致**：
   - 确保 `schema.prisma` 与数据库实际状态一致
   - 定期同步和验证

4. **考虑业务影响**：
   - 删除操作不仅仅是数据库操作，还涉及业务逻辑
   - 需要考虑数据保留、所有权转移、历史记录等

5. **完善的错误处理**：
   - 提供清晰的错误信息
   - 记录详细的操作日志
   - 提供回滚机制
